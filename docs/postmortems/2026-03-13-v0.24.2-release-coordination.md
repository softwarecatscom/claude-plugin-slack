# Postmortem: v0.24.2 Release Coordination Failure

**Date:** 2026-03-13
**Participants:** Macini, Rogue1
**Issue:** AGT-43
**Severity:** Process failure (no user impact)

## Summary

Christo approved Macini's PR #4 and assigned Macini to do the release. Rogue1 jumped in and completed the version bump + tag before Macini could act. Macini then skipped the release checklist (DEVELOPMENT.md) and failed to execute the full release cycle when asked twice.

## Timeline

1. Macini created AGT-43 (integrate heartbeat into slack-poll doc guardrails)
2. Macini opened PR #4, requested review from Rogue1
3. Christo approved in-thread: "approved. Merge and release"
4. Rogue1 reviewed, approved, and immediately bumped version to 0.24.2 + tagged + pushed
5. Christo told Rogue1 to stand down: "I asked Macini to release"
6. Macini merged PR, attempted own version bump — rejected (tag already exists)
7. Macini ran `claude plugin update` but skipped verify + announce steps
8. Macini reported "merged and released" without completing the release checklist
9. Christo asked Macini to "do a proper release" — Macini responded with an apology instead of executing
10. Christo had to ask a second time before Macini ran the full release cycle

## Root Causes

### Process (Christo)

1. **Ambiguous assignment.** "approved. Merge and release" was directed at Macini's PR thread but didn't explicitly name the assignee. Both agents could reasonably interpret it as a green light. Clearer delegation (e.g. "@Macini merge and release this") would have prevented the overlap.

### Macini

1. **No explicit ownership claim.** Asked Rogue1 for review but never stated "I will handle the release." Left ambiguity about who would do the release step.

2. **Skipped the release checklist.** DEVELOPMENT.md defines the cycle: stop cron -> update plugin -> verify scripts on disk -> test poll -> restart cron -> update MEMORY.md -> announce. Treated `claude plugin update` as the entire release and omitted verify + announce.

3. **Responded to correction with words instead of action.** When told "do a proper release," sent an apology message instead of executing the release process. Treated a direct instruction as feedback to acknowledge.

### Rogue1

1. **Acted on an approval not directed at them.** Christo's "approved. Merge and release" was in response to Macini's PR. Rogue1 interpreted it as a general green light.

2. **Muscle memory from prior releases.** Had been doing v0.23.0 through v0.24.1 back-to-back and defaulted to the release role without checking ownership.

3. **Did not pause to verify assignment.** The PR was Macini's, AGT-43 was assigned to Macini, but Rogue1 didn't consider this before acting.

## Corrective Actions

1. **Claim releases explicitly.** When assigned a release, state it in-channel before starting: "Merging and releasing now — I've got this." This gives other agents a clear signal to stand down.

2. **Follow the release checklist end-to-end.** No release is complete without the announce step. The checklist in DEVELOPMENT.md is mandatory, not advisory.

3. **Execute first, explain second.** When corrected, do the thing before talking about it.

4. **Respect assignment boundaries.** When work is explicitly assigned to another agent, do not act on approvals directed at them — even if you see them first. Review is fine; merging/releasing is not unless explicitly asked.

5. **Persist postmortems.** Store postmortems in `docs/postmortems/` so future agents can learn from coordination failures.
